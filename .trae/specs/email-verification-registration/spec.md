# 邮箱验证码注册功能 - 产品需求文档 (PRD)

## 一、项目概述

### 1.1 项目背景
当前校园论坛系统的注册流程缺少邮箱验证环节，存在以下问题：
- 用户可能填写错误邮箱
- 缺乏身份验证机制
- 无法通过邮箱找回密码

### 1.2 项目目标
在注册流程中新增邮箱验证码验证功能，提高账号安全性。

### 1.3 目标用户
- 新注册用户
- 系统管理员

## 二、功能需求

### 2.1 发送验证码
**用户故事**：作为新用户，我希望在注册时点击"发送验证码"按钮，向我的邮箱发送一封包含验证码的邮件，以便验证邮箱的有效性。

**功能描述**：
- 用户填写邮箱后可点击"发送验证码"
- 系统生成6位数字验证码
- 验证码有效期5分钟
- 同一邮箱发送间隔不小于60秒
- 同一邮箱每日最多发送10次

**输入**：
- 邮箱地址（必填）
- 验证类型（固定为 "REGISTER"）

**输出**：
- 成功提示："验证码已发送，请查收邮件"
- 失败提示："发送太频繁，请60秒后再试" 或其他错误信息

### 2.2 验证验证码
**用户故事**：作为新用户，我希望输入邮箱验证码后可以完成注册，以便验证我的身份。

**功能描述**：
- 验证验证码是否正确
- 验证验证码是否在有效期内
- 验证验证码是否已被使用
- 验证码错误次数最多5次
- 验证成功后标记验证码已使用

**输入**：
- 邮箱地址
- 验证码
- 验证类型

**输出**：
- 成功/失败状态
- 错误信息

### 2.3 注册流程改造
**用户故事**：作为新用户，我希望填写邮箱验证码后才能完成注册，以便确保我是邮箱的所有者。

**功能描述**：
- 注册时必须验证邮箱验证码
- 验证通过后才创建用户账号
- 验证失败时显示相应错误提示

**输入**：
- 学号
- 昵称
- 邮箱
- 密码
- 确认密码
- 验证码

**输出**：
- 注册成功/失败状态
- JWT token（成功时）

## 三、非功能需求

### 3.1 性能要求
- 邮件发送响应时间 < 5秒
- 验证码验证响应时间 < 100ms

### 3.2 安全要求
- 验证码使用安全随机数生成
- 验证码不能在日志中明文显示
- 防止邮箱枚举（不提示"邮箱已注册"，使用通用错误）
- 防止暴力破解（限制重试次数）

### 3.3 可用性要求
- 界面友好，操作简单
- 提供清晰的错误提示
- 发送验证码后有倒计时提示

## 五、数据设计

### 5.1 数据库表

#### 表名：`email_verification`

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | BIGINT | PRIMARY KEY, AUTO_INCREMENT | 主键ID |
| email | VARCHAR(100) | NOT NULL | 邮箱地址 |
| verification_code | VARCHAR(10) | NOT NULL | 验证码 |
| verification_type | VARCHAR(20) | NOT NULL, DEFAULT 'REGISTER' | 验证类型 |
| send_count | INT | NOT NULL, DEFAULT 1 | 发送次数 |
| verify_count | INT | NOT NULL, DEFAULT 0 | 验证尝试次数 |
| is_verified | TINYINT(1) | NOT NULL, DEFAULT 0 | 是否已验证：0-未验证，1-已验证 |
| expire_time | DATETIME | NOT NULL | 过期时间 |
| create_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**索引**：
- UNIQUE INDEX `idx_email_type` (`email`, `verification_type`)
- INDEX `idx_expire_time` (`expire_time`)
- INDEX `idx_create_time` (`create_time`)

### 5.2 API设计

#### 接口1：发送邮箱验证码
- **URL**：`POST /api/auth/sendEmailCode`
- **请求参数**：
  ```json
  {
    "email": "user@example.com",
    "type": "REGISTER"
  }
  ```
- **响应**：
  ```json
  {
    "code": 0,
    "msg": "验证码已发送，请查收邮件",
    "data": null
  }
  ```

#### 接口2：验证邮箱验证码（可选，实际在注册时验证）
- **URL**：`POST /api/auth/verifyEmailCode`
- **请求参数**：
  ```json
  {
    "email": "user@example.com",
    "code": "123456",
    "type": "REGISTER"
  }
  ```
- **响应**：
  ```json
  {
    "code": 0,
    "msg": "验证成功",
    "data": null
  }
  ```

#### 接口3：注册用户（改造）
- **URL**：`POST /api/auth/registerUser`
- **请求参数**（新增验证码字段）：
  ```json
  {
    "username": "202100123456",
    "password": "123456",
    "perName": "张三",
    "email": "user@example.com",
    "emailCode": "123456"
  }
  ```
- **响应**（保持现有格式）：
  ```json
  {
    "code": 0,
    "msg": "注册成功！",
    "data": null
  }
  ```

## 七、业务规则

### 7.1 验证码规则
- 验证码长度：6位纯数字
- 验证码有效期：5分钟（300秒）
- 验证码重试次数：最多5次
- 发送间隔：同一邮箱至少60秒
- 每日发送上限：同一邮箱最多10次

### 7.2 验证规则
- 验证码必须在有效期内
- 验证码必须未被使用过
- 验证码必须与邮箱匹配
- 验证成功后立即标记为已使用

### 7.3 防刷规则
- 同一邮箱60秒内只能发送1次
- 同一邮箱每日最多发送10次
- 验证码错误5次后需重新获取

## 八、错误处理

### 8.1 错误码定义

| 错误场景 | 错误提示 |
|----------|----------|
| 邮箱格式不正确 | "邮箱格式不正确" |
| 发送太频繁 | "发送太频繁，请60秒后再试" |
| 今日发送次数已达上限 | "今日发送次数已达上限，请明日再试" |
| 邮件发送失败 | "邮件发送失败，请稍后重试" |
| 验证码已过期 | "验证码已过期，请重新获取" |
| 验证码错误 | "验证码错误，请重新输入" |
| 验证码已被使用 | "验证码已被使用，请重新获取" |
| 验证次数已达上限 | "验证次数过多，请重新获取验证码" |

## 九、验收标准

### 9.1 功能验收
- [ ] 用户可以成功发送邮箱验证码
- [ ] 用户可以通过验证码完成注册
- [ ] 验证码过期后无法使用
- [ ] 验证码错误时有清晰提示
- [ ] 发送频率限制生效
- [ ] 验证次数限制生效

### 9.2 性能验收
- [ ] 邮件发送响应时间 < 5秒
- [ ] 验证码验证响应时间 < 100ms

### 9.3 安全验收
- [ ] 验证码使用安全随机数生成
- [ ] 日志中不显示明文验证码
- [ ] 无法通过错误提示枚举邮箱

## 十、风险与应对

### 10.1 技术风险
| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| QQ邮箱SMTP配置失败 | 高 | 中 | 提前测试，准备备用邮箱 |
| 邮件发送延迟 | 中 | 中 | 提示用户可能延迟，检查垃圾箱 |
| 邮件进入垃圾箱 | 低 | 中 | 优化邮件内容，添加提示 |

### 10.2 业务风险
| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 用户收不到邮件 | 中 | 低 | 提供客服支持，添加说明 |
| 恶意发送邮件 | 中 | 中 | 完善防刷机制 |
